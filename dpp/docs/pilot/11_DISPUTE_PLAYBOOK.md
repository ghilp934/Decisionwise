# Dispute Playbook

**Decisionproof Paid Pilot - 분쟁 대응 플레이북 (파일럿용)**

문서 버전: 2026-02-16

---

## 목적

본 문서는 유료 파일럿 단계에서 발생할 수 있는 **과금/정산 분쟁**, **장애/가용성 분쟁**, **보안 사고 관련 분쟁**을 **빠르게 수습**하기 위한 최소 절차(결정론)를 제공합니다.

- 원칙: “의견”보다 **증거(ledger/log/request_id)** 우선
- 목표: 분쟁 장기화 방지, 반복 원인 제거(재발 방지)

## 1) 분쟁 범위

### A. 과금/정산 분쟁
- “실제 사용량 vs 청구액” 불일치
- 초과 사용 차단/추가 과금의 적용 여부

### B. 장애/가용성 분쟁
- 서비스 중단/오류로 인한 업무 영향
- SLA-lite 범위 내 보상(크레딧 등) 논의

### C. 보안/사고 분쟁
- 침해 의심/보안 사고 발생 시 통지/조치
- 로그/증적 제공 범위

## 2) 공통 원칙 (SSOT)

1) **접수 즉시 티켓 발급**: 모든 분쟁은 티켓 시스템(또는 이메일 스레드)으로만 처리합니다.
2) **증거 우선순위**: Usage Ledger ID / Request ID / Server logs(요약) / Billing statement 순
3) **시간 기준(목표)**: 접수 확인 2시간 이내, 1차 사실관계 정리 1영업일 이내
4) **개인정보 최소화**: 분쟁 증거 제출 시 원문 데이터 대신 ID/해시/요약을 우선합니다.

## 4) 분쟁 처리 단계

### Step 1. 확인(Acknowledge)
- Decisionproof가 접수 확인 회신 (목표: 2시간)
- 담당자 지정 및 다음 업데이트 시간 공지

### Step 2. 사실관계 정리(Establish Facts)
- Decisionproof는 아래를 기준으로 사실관계를 정리합니다.
  - 계량 로그/원장(ledger)
  - 에러 응답(application/problem+json) 및 request_id
  - 시스템 이벤트(배포, 장애, 제한 정책 변경)

### Step 3. 원인 분류(Root Cause Tagging)
- 과금: 계량/쿼터/차단 룰, 이중 차감, 환불/크레딧 처리 누락
- 장애: 배포/인프라/인증/외부 의존성
- 보안: 오탐/침해/오남용(AUP 위반 포함)

### Step 4. 구제(Remedy) 선택
- 과금 분쟁: (a) 정정 청구서 (b) 크레딧 지급 (c) 환불 (d) 혼합
- 장애 분쟁: (a) 크레딧 (b) 기간 연장 (c) 환불(예외)
- 보안 분쟁: (a) 토큰 회수/재발급 (b) 접근 차단 (c) 고객 통지(필요 시)

> 기본값: 파일럿 단계에서는 “현금 환불”보다 “크레딧/기간 연장”을 우선 고려합니다.

### Step 5. 종결(Close) 및 재발 방지
- 합의 사항을 **서면(티켓/이메일)**으로 확정
- 재발 방지 항목(설정/문서/코드) 1개 이상 도출
- Changelog 반영(`09_CHANGELOG_AND_CONTACTS.md`)

---

## 5) 증거(Artifacts) 목록

- Usage Ledger: X-DPP-Usage-Ledger-Id, (또는 동등한 내부 원장 ID)
- Request IDs: request_id / trace_id (가능하면 고객이 제출)
- Billing artifacts: 청구서/세금계산서/결제 영수증
- Incident artifacts: 사건 타임라인, 영향 범위, 완화 조치
