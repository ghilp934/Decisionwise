## 2. Migrations: Files + DDL Summary + Safety

### File 1: `20260218_backfill_user_tenants_p0_3_1.sql` (NEW - 234 lines)

**Purpose**: Backfill personal tenants and user-tenant mappings for existing users

**DDL Summary**:
```sql
-- STEP 1: Create personal tenant for each auth.users record
INSERT INTO tenants (tenant_id, display_name, status, created_at)
SELECT
    'user_' || substring(au.id::text, 1, 8) AS tenant_id,
    COALESCE(au.email, 'User ' || substring(au.id::text, 1, 8)) AS display_name,
    'ACTIVE' AS status,
    NOW() AS created_at
FROM auth.users au
WHERE au.deleted_at IS NULL
  AND au.banned_until IS NULL
ON CONFLICT (tenant_id) DO NOTHING;  -- Idempotent

-- STEP 2: Create user-tenant mapping (role=owner)
INSERT INTO user_tenants (id, user_id, tenant_id, role, status)
SELECT
    gen_random_uuid(),
    au.id,
    'user_' || substring(au.id::text, 1, 8),
    'owner',
    'active'
FROM auth.users au
WHERE au.deleted_at IS NULL
  AND au.banned_until IS NULL
ON CONFLICT (user_id, tenant_id) DO NOTHING;  -- Idempotent
```

**Safety Features**:
- ✅ **Idempotent**: `ON CONFLICT DO NOTHING` - safe to re-run
- ✅ **No deletes**: Only INSERT operations (non-destructive)
- ✅ **Inline verification**: Checks for orphans, duplicates, null values
- ✅ **Summary report**: Logs counts and status at end
- ✅ **Preserves existing data**: Non-personal tenants (system/test) kept intact

**Why Safe**:
1. Uses UNIQUE constraint `(user_id, tenant_id)` to prevent duplicates
2. Deterministic tenant_id generation: `user_<first_8_chars_of_user_id>`
3. Only processes active users (excludes deleted/banned)
4. Does NOT modify or delete existing `tenants` table records
5. Rollback-safe: No cascading deletes or destructive updates

**Expected Result** (for 10 Pilot users):
- 10 new tenants created (tenant_id format: `user_a1b2c3d4`, etc.)
- 10 user_tenant mappings created (role=owner, status=active)
- Existing non-personal tenants preserved (if any)

**Changes**:
- Added DB dependency: `from dpp_api.db.session import get_db`
- Added models: `from dpp_api.db.models import Tenant, UserTenant`
- Modified `signup()` function signature: `async def signup(request: SignupRequest, db: Session = Depends(get_db))`
- **Added tenant creation logic** (lines 118-151):

```python
# P0-3.1: Auto-create personal tenant and user-tenant mapping
try:
    # Generate deterministic tenant_id from user_id
    tenant_id = f"user_{user_id[:8]}"

    # Ensure unique tenant_id (collision protection)
    counter = 0
    while db.query(Tenant).filter(Tenant.tenant_id == tenant_id).first():
        counter += 1
        tenant_id = f"user_{user_id[:8]}_{counter}"

    # Create personal tenant
    new_tenant = Tenant(
        tenant_id=tenant_id,
        display_name=user_email,
        status="ACTIVE",
    )
    db.add(new_tenant)

    # Create user-tenant mapping as owner
    user_tenant = UserTenant(
        id=str(uuid.uuid4()),
        user_id=user_id,
        tenant_id=tenant_id,
        role="owner",
        status="active",
    )
    db.add(user_tenant)

    db.commit()
except Exception as tenant_error:
    db.rollback()
    logger.error("auth.signup.tenant_creation_failed", ...)
    # Don't fail signup - degraded state but user can login
```

**Why Safe**:
- ✅ Deterministic tenant_id (same as migration)
- ✅ Collision protection (while loop checks existing)
- ✅ Graceful degradation (tenant creation failure doesn't block signup)
- ✅ Logging for debugging
- ⚠️ **Degraded state handling**: If tenant creation fails, user exists in auth.users but has no tenant
  - **Mitigation**: Manual intervention or re-run backfill migration

## 3. Backfill Source-of-Truth Decision

### Chosen Strategy: **auth.users as Authoritative Source**

**Rationale**:
1. **No guessing**: auth.users (Supabase) is the only reliable user list
2. **tenants table lacks owner info**: No `owner_user_id` or `created_by` column
3. **API keys lack user tracking**: No `created_by_user_id` column
4. **Pilot assumption**: Each user should have exactly 1 personal tenant (1:1 mapping)

**Alternative Sources Rejected**:
- ❌ **tenants table**: No way to know which user owns which tenant
- ❌ **API keys**: No `created_by` field, multiple keys can exist per tenant
- ❌ **Manual CSV**: Not needed - can derive from auth.users deterministically

**Mapping Rule**:
```
FOR EACH user IN auth.users WHERE deleted_at IS NULL:
    tenant_id = "user_<first_8_chars_of_user.id>"
    role = "owner"
    status = "active"
```

**Edge Cases Handled**:
1. **Collision**: If `user_a1b2c3d4` already exists, append suffix `_1`, `_2`, etc.
2. **Soft-deleted users**: Excluded (WHERE deleted_at IS NULL)
3. **Banned users**: Excluded (WHERE banned_until IS NULL)
4. **Existing non-personal tenants**: Preserved (no owner assigned, manual intervention needed)
